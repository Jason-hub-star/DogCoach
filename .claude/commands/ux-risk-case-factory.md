# /ux-risk-case-factory — UX 위험 케이스 대량 생성/병렬 검증

실사용자가 겪기 쉬운 불쾌한 경험을 미리 많이 만들어서, 병렬로 테스트하고 우선순위를 정리하는 명령 문서다.

## 사용 시점
- “이 플로우가 실제로 불편한가?”를 빠르게 넓게 보고 싶을 때
- 로그인/온보딩/설문/대시보드/기록/코치/설정 흐름의 빈틈을 찾고 싶을 때
- 하나의 문제를 고치는 김에 주변 UX 리스크까지 같이 줄이고 싶을 때

## 목표
- 한 번에 많은 실패 사례를 만든다.
- 서브에이전트에 분산해서 병렬로 조사한다.
- 중복을 제거하고, 실제 사용자 피해가 큰 것부터 정렬한다.

## 병렬 실행 전략
- 공통 입력 고정: 같은 서비스 URL, 같은 계정 유형, 같은 디바이스 범주, 같은 시간대 조건을 준다.
- 서브에이전트 분배: A는 게스트 온보딩/설문 실패, B는 로그인 후 세션 유지/재진입, C는 대시보드 탭 이동/빈 상태/로딩 지연, D는 네트워크 오류/재시도/중복 제출, E는 모바일 화면/짧은 viewport/스크롤 꼬임을 본다.
- 출력 형식 통일: 모든 에이전트가 같은 표로 반환한다.
- 합치기: 중복 제거, 실제 재현 가능성 확인, 사용자 고통도 기준으로 우선순위 매기기를 한 번에 수행한다.

## 케이스 생성 축
- 인증: 게스트, 로그인, 세션 만료, 재로그인
- 온보딩: survey 중단, 되돌아가기, 중복 제출, 입력 누락
- 결과: 제출 후 result 미표시, 지연 로딩, 잘못된 리다이렉트
- 대시보드: 탭 전환, 로딩 스켈레톤 장기화, 데이터 없음, 부분 데이터
- 기록/코치/설정: 저장 실패, 빈 값, 권한 오류, 업로드 실패
- 런타임: 느린 API, 404, 500, CORS처럼 보이는 실패, 새로고침

## 바로 쓸 수 있는 서브에이전트 프롬프트
```text
너는 UX 리스크 탐지 서브에이전트다.
대상 경로: 랜딩 -> login/survey -> result -> dashboard -> log -> coach -> settings
목표: 실제 사용자에게 안좋은 경험이 되는 실패 사례를 10개 이상 만든다.
조건:
- guest 흐름과 logged-in 흐름을 둘 다 본다.
- API 직접 실패와 browser E2E 실패를 분리해서 적는다.
- request_id가 있으면 반드시 적는다.
출력:
- case_id
- 경로
- 재현 단계
- 기대 결과
- 실제 결과
- 리스크 요약
- 우선순위
```

## 수집 템플릿
```text
| case_id | flow | risk | repro | expected | actual | evidence | priority |
|---|---|---|---|---|---|---|---|
| UX-001 | guest | survey submit fail | ... | ... | ... | screenshot/log | P0 |
```

## 합격 기준
- 단순히 “깨졌다”가 아니라, 왜 사용자에게 나쁜지 설명할 수 있어야 한다.
- 재현 가능해야 한다.
- API 문제인지 브라우저 문제인지 나눠야 한다.
- 같은 근본 원인에서 파생되는 케이스를 묶어야 한다.

## 추천 운영 순서
1. 서브에이전트로 케이스 20~50개 생성
2. 중복/노이즈 제거
3. 실제 재현 가능한 것만 선별
4. P0/P1부터 순서대로 수정
5. 수정 후 guest + logged-in + browser E2E + API direct를 다시 돌린다
